## Назначение проекта

Cliniss - сайт, который будет разрабатываться на Python Django с использованием SQLite в качестве базы данных для локальной разработки и production.

Текущий репозиторий содержит статический экспорт сайта в папке `cliniss/`. Этот экспорт рассматривается как исходный материал для переноса в Django: структура страниц, тексты, изображения, стили, скрипты, `robots.txt`, `sitemap.xml` и другие статические артефакты должны учитываться при проектировании будущего Django-приложения.

## Технологический стек

- Backend: Python 3.14.5, Django 6.0.6.
- Database: SQLite (`db.sqlite3`).
- Frontend: Django templates, HTML, CSS, JavaScript.
- Static/media assets: новая статика в `static/`, пользовательские медиа в `media/`, исходные материалы Tilda в `cliniss/`.

## Целевая архитектура

Базовый вариант: обычный Django-проект без CMS. Контент редактируется разработчиком через VS Code, поэтому основная логика страниц, текстов и структуры должна быть понятной в коде, без визуального конструктора и тяжелых CMS-зависимостей.

Рекомендуемые приложения:

- `core` - настройки сайта, общие layout-шаблоны, меню, контакты, SEO, sitemap, robots.
- `pages` - информационные страницы, юридические страницы, статичные разделы, поиск, sitemap, robots.
- `content` - будущие структурированные данные для сотрудников, направлений работ, услуг, документов, галерей, FAQ, исследований.
- `leads` - все формы заявок, хранение обращений, уведомления, антиспам, согласия на обработку персональных данных.
- `accessibility` - клиентская версия для слабовидящих и настройки доступности, если не будет вынесено в общий frontend-код.

SQLite используется и для локальной разработки, и для production.

Заявки на первом этапе должны проектироваться так, чтобы их можно было хранить локально и позже связать с другим сайтом, который планируется объединить с этим проектом. До уточнения интеграции нельзя жестко завязывать формы на внешнюю CRM или одноразовый транспорт.

Английская версия должна стать полной копией русской по структуре и ключевым страницам, включая аналоги `/volunteers`, `/docs`, `/policy`, `/agreement` и других публичных разделов.

Дизайн можно аккуратно переработать под более чистую, современную и адаптивную верстку. Требования: сохранить узнаваемость CLINISS, контент, основные сценарии и SEO-URL, но не копировать ограничения Tilda Zero Block.

## Текущая структура Django

- `config/settings.py` - настройки проекта, SQLite, шаблоны, статика, локаль `ru`, часовой пояс `Europe/Moscow`.
- `config/site_bundle_settings.py` - настройки для exe-бандла сайта: наследует основные настройки, включает локальный `DEBUG`, ограничивает hosts на `127.0.0.1`/`localhost`, использует writable data dir рядом с exe для SQLite и media.
- `config/test_bundle_settings.py` - настройки для exe-бандла тестов: наследует основные настройки и использует SQLite in-memory, чтобы тестовый запуск не требовал внешней базы.
- `config/urls.py` - подключение `admin/`, `leads.urls`, `pages.urls`.
- `core/site.py` - кодовые настройки сайта: контакты, навигация RU/EN, социальные ссылки.
- `core/context_processors.py` - общий контекст для шаблонов.
- `pages/data.py` - кодовое хранилище страниц RU/EN, пар языковых версий, карточек, FAQ, ссылок, hero-изображений, галерей, карт, фактов, настроек форм и ссылок на Яндекс Карты.
- `pages/data.py` поддерживает расширенный формат секций: `body`, `bullets`, `facts`, `cards`, `faq`, `links`, `image`, `images`, `gallery_style`, `map`, `stats`, `lead_form`, `cta_label`, `cta_url`; ключ `items` не используется из-за конфликта с методом словаря в Django templates.
- `pages/views.py` - вывод страниц, поиск, `robots.txt`, `sitemap.xml`.
- `leads/models.py` - модель заявки `Lead`.
- `leads/forms.py` - Django-форма заявки.
- `leads/views.py` - обработка POST `/leads/submit`.
- `templates/base.html` - общий layout, меню с состояниями раскрытия групп, footer, cookie-баннер, панель доступности.
- `templates/pages/detail.html` - общий шаблон информационной страницы; поддерживает hero, факты, `bullets`, карточки, FAQ, ссылки, секционные CTA-кнопки, одиночные изображения, карты и галереи с вариантом отображения документов.
- `templates/pages/search.html` - страница поиска.
- `templates/includes/lead_form.html` - форма заявки; набор типов заявки, заголовок и выбранный вариант берутся из `page.lead_form`.
- `static/site/css/main.css` - новая адаптивная верстка, включая hero-блоки, карточки, адаптивные плитки списков и фактов, переносы длинного текста, геоблок карты, ограничение размеров фото, галереи фото и документные галереи без обрезки сканов.
- `static/site/js/main.js` - мобильное меню и раскрытие его групп, cookie-согласие, клиентские настройки доступности и миграция старого состояния скрытия изображений.
- `static/site/img/photos/` - отобранные фото из Tilda-экспорта с семантичными именами.
- `static/site/img/team/` - портреты сотрудников.
- `static/site/img/documents/` - изображения документов: сканы лицензии и организационная структура.
- `static/site/img/graphics/` - фирменные баннеры и графические обложки из Tilda-экспорта.
- `scripts/run_test_bundle.py` - точка входа PyInstaller-бандла, по умолчанию запускает `django test`, пишет лог рядом с exe и удерживает окно открытым при запуске из Проводника.
- `scripts/build_test_exe.ps1` - PowerShell-скрипт установки build-зависимостей и сборки `dist/cliniss-tests.exe`.
- `scripts/run_site_bundle.py` - точка входа PyInstaller-бандла сайта: выбирает свободный порт, применяет миграции, запускает Django `runserver --noreload`, открывает браузер и пишет лог.

- `cliniss_test.spec` - спецификация PyInstaller, включающая Django-приложения, шаблоны и статику.

## Модель данных

### `leads.Lead`

Хранит заявки сайта в SQLite.

Поля:

- `created_at`, `updated_at`;
- `status`: новая, в работе, закрыта, архив;
- `lead_type`: звонок, команда, коммерческое предложение, доброволец, отзыв, исследование, вопрос;
- `language`: `ru` или `en`;
- `name`, `phone`, `email`, `message`;
- `source_path`, `source_site`;
- `consent`;
- `integration_key`, `external_payload` для будущей связи с другим сайтом;
- `user_agent`, `ip_address`.

## Команды разработки

Установка зависимостей:

```powershell
.\.venv\Scripts\python -m pip install -r requirements.txt
```

Запуск локального сервера:

```powershell
.\.venv\Scripts\python manage.py runserver 127.0.0.1:8000
```

Команды Django выполняются из корня проекта `C:\Codex\Cliniss`. Папка `cliniss/` - это исходный Tilda-экспорт, в ней нет `manage.py`; запускать сервер из нее напрямую не нужно.

Миграции:

```powershell
.\.venv\Scripts\python manage.py makemigrations
.\.venv\Scripts\python manage.py migrate
```

Проверки:

```powershell
.\.venv\Scripts\python manage.py check
.\.venv\Scripts\python manage.py test
```

Сборка exe для запуска тестов на Windows-ПК без Python:

```powershell
.\scripts\build_test_exe.ps1
```

Готовый файл появляется в `dist/cliniss-tests.exe`. При запуске без аргументов exe выполняет проектные Django-тесты приложений `core`, `pages`, `content`, `leads`. Можно передавать аргументы Django test runner, например:

```powershell
.\dist\cliniss-tests.exe pages
.\dist\cliniss-tests.exe test leads -v 2
.\dist\cliniss-tests.exe check
```

При запуске двойным кликом окно остается открытым до нажатия Enter. Вывод последнего запуска сохраняется в `cliniss-tests.log` рядом с exe.

Сборка exe для запуска локального сайта на Windows-ПК без Python:

```powershell
.\scripts\build_site_exe.ps1
```

Готовый файл появляется в `dist/cliniss-site.exe`. При запуске exe применяет миграции, поднимает локальный сервер и открывает сайт в браузере. Основной адрес:

```text
http://127.0.0.1:8000/
```

Если порт `8000` занят, exe автоматически выберет следующий свободный порт и напечатает фактический адрес в окне и в `cliniss-site.log`. Порт можно задать вручную переменной окружения `CLINISS_SITE_PORT`, отключить автозапуск браузера - `CLINISS_SITE_NO_BROWSER=1`.

Зависимости для сборки exe зафиксированы отдельно в `requirements-build.txt`; runtime-зависимости сайта остаются в `requirements.txt`.

## Не переносить из Tilda напрямую

- Не копировать `page*.html` как финальные Django-шаблоны без переработки: в них много Tilda Zero Block, абсолютного позиционирования, инлайн-стилей, служебных скриптов и повторяющихся блоков.
- Не оставлять Tilda-формы: их нужно заменить на Django forms/views/models.
- Не тащить все JS/CSS Tilda как постоянную зависимость, кроме временного переходного этапа.
- Не делать `header` и `footer` отдельными публичными страницами.

## Известные проблемы и вопросы миграции

- В английских страницах есть ссылка на `/policy-en`, но такой страницы нет в `sitemap.xml` и в локальных исходниках. При портировании нужно создать полноценную английскую версию политики.
- Нужно спроектировать заявки с учетом будущего объединения с другим сайтом.
- Нужно отдельно проверить требования к хранению персональных данных и согласий, так как формы собирают имя, телефон, email, резюме, отзывы и данные добровольцев.
- Юридические страницы `/policy`, `/agreement` и их EN-аналоги расширены до более полной структурированной редакции с разделами о понятиях, целях, правах, принципах, условиях обработки, сроке действия согласия, cookie и контактах. Перед production все равно нужна отдельная юридическая сверка с полным текстом Tilda-страниц и актуальными требованиями 152-ФЗ.
- Английские страницы, которых не было в старом `sitemap.xml` (`/areaswork-en`, `/volunteers-en`, `/docs-en`, `/issledovaniya-bioekvivalentnosti-en`, `/policy-en`, `/agreement-en`), заполнены как рабочие EN-аналоги, но требуют редакторской/юридической вычитки.
